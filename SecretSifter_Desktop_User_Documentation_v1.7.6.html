1. Executive Summary

SecretSifter Desktop is a Windows-native, single-installer secret scanner for live web applications. It ships with the same detection engine as the Burp extension edition, plus a Playwright-driven crawler, MITM proxy, and an automation API — no Burp license required.

The tool is designed for security analysts who need to discover hard-coded credentials in modern single-page applications, API responses, request headers, and proxied traffic. It does not require the analyst to be running a paid web-proxy product. Findings appear in a Burp-style dashboard with severity badges, sortable columns, and a Details / Request / Response side panel for each finding.

What's New in v1.7.6

  • Mask Value / Mask URL fixed (now reliably masks the right column even after column reordering).
  • Start Scan auto-clears prior results and dedup state — no more stale findings bleeding into a fresh run.
  • Compact 📋 copy icons in the Details panel free up screen real estate for long URLs.
  • Center divider has hard min-widths so dragging can no longer hide either pane.
  • New AI Provider API Key field (Anthropic / OpenAI / etc.) with masked input and 👁 reveal toggle, persisted to local user prefs.

3. Features Overview

Bulk URL Scan

Paste a list of URLs, optionally enable Playwright browser-mode for SPAs. Each URL is fetched (with auth-failure detection), all referenced JS bundles are followed, and every response body is run through the full detection pipeline.

Playwright Crawler

Drives real Google Chrome (auto-detected) via a stealth-flagged Playwright session. Captures every request and response, follows in-app navigation, discovers SPA-loaded resources that bulk fetch alone misses.

MITM Proxy

Embedded HTTP/1.1 proxy with on-the-fly per-host TLS certificates (Bouncy Castle CA). Point any browser at 127.0.0.1:8888, import the exported CA, and inflight traffic is scanned and surfaced as findings.

HAR Import

Replay any browser DevTools HAR file through the scanner. All requests + responses are restored to the capture store as if the proxy had captured them, so the Details panel works the same way.

Findings Dashboard

Sortable / filterable table with severity badges, NOISE triage state, mask-value / mask-URL toggles, column visibility menu, and a Burp-style Details / Request / Response side panel.

Custom Rules

User-defined regex rules (RuleName | regex | severity). Run additively with built-in detection, or in raw mode alongside the "Use ONLY" toggle that bypasses every default noise filter.

Reports & Exports

All-in-One Report ZIP (HTML + CSV + JSON + suppressed CSV) and Per-Domain Reports ZIP. Every artifact timestamped yyyyMMdd_HHmmss. CSV columns match the HTML report.

REST API + MCP

Bearer-authenticated JSON-RPC + REST server on localhost. Lets Claude Desktop / Claude Code CLI / curl / Python automate scans, fetch findings, update settings.

4.1 Prerequisites

4.2 Install

  1. Download SecretSifter-1.7.6.exe.
  2. Double-click. Windows SmartScreen may warn — click More info → Run anyway.
  3. Follow the installer prompts. Default install path is %LOCALAPPDATA%\SecretSifter.
  4. The installer creates a Start Menu shortcut and (optional) Desktop icon.

4.3 First Run

On first launch, SecretSifter:

  • Generates a random MCP port (8000–9000 range) and a 32-hex Bearer token. Both visible in Settings → AI / MCP.
  • Defaults to FULL tier scanning, entropy threshold 3.5, PII detection enabled, request-header scanning enabled.
  • Loads the default key blocklist (STORAGE_KEY_, STATE_KEY_, NEXT_PUBLIC_, REACT_APP_PUBLIC_, VUE_APP_PUBLIC_).
  • Settings persisted to the Windows registry under HKCU\Software\JavaSoft\Prefs\com\secretsifter.

5. Usage Guide

5.1 Bulk URL Scan

  1. Open the Scan tab.
  2. Paste one URL per line into the Target URLs area.
  3. (Optional) Enable Browser Mode for Playwright crawling — captures SPA-loaded resources.
  4. Set Threads if you want parallel fetching (default 4).
  5. Click Start Scan. Findings stream into the table as they're discovered.
  6. Click any row to populate the Details / Request / Response tabs.
Re-running a scan: v1.7.6 auto-clears previous findings, dedup state, and the capture store when you click Start Scan again. Earlier versions appended to the prior run, which silently suppressed duplicates.

5.2 MITM Proxy

  1. Settings → Proxy → Start Proxy. Default port is 8888.
  2. Click Export CA Certificate… and save SecretSifterCA.crt.
  3. Import the .crt into your browser's trusted root authority store:
    • Chrome / Edge: Settings → Privacy & security → Security → Manage certificates → Authorities tab → Import.
    • Firefox: Settings → Privacy & Security → View Certificates → Authorities → Import.
  4. Configure your browser's HTTP & HTTPS proxy to 127.0.0.1:8888 (no comma in the port).
  5. Browse normally. Each intercepted request is logged in the Logs tab as [Proxy] METHOD url. Findings flow into the Scan tab.
HSTS sites: some banks / cloud consoles refuse non-trusted CAs even after manual import. You'll see SSL errors that can't be bypassed; these hosts are out of scope for the proxy. Use the bulk scanner or HAR import instead.

5.3 HAR Import

  1. In Chrome / Firefox, open DevTools → Network tab. Browse the target session.
  2. Right-click in the request list → Save all as HAR with content.
  3. In SecretSifter: Scan → Import HAR… → select the .har file.
  4. Every request / response in the HAR is restored to the capture store and run through the scanner. The Details panel works the same as if the proxy had captured them.

5.4 Custom Rules

  1. Settings → Custom Rules.
  2. One rule per line, format: RuleName | regex | severity where severity ∈ HIGH / MEDIUM / LOW / INFORMATION.
  3. Lines starting with # are comments.
  4. Enable custom rules — adds your rules on top of built-in detection.
  5. Use custom rules ONLY — runs only your regexes; bypasses every default noise filter (raw mode).
# Example custom rules
TelegramBotToken | \b[0-9]{5,16}:A[0-9A-Za-z_-]{34}\b      | HIGH
HuggingFaceToken | \bhf_[A-Za-z0-9]{34}\b                  | HIGH
InternalToken    | INT-[0-9]{8}-[A-Z]{4}                   | HIGH
ServiceAccount   | sa-[a-z0-9]{16}@internal\.example\.com  | MEDIUM

5.5 Triage

  1. Click a row in the findings table.
  2. The Details tab shows Rule ID / Name, Key, Line, Source URL, Target URL, Matched Value (with 📋 copy), and Context.
  3. Adjust Severity or Confidence via the dropdowns — changes are reflected in the table immediately.
  4. Mark a finding as NOISE by setting either Severity or Confidence to NOISE; it's then excluded from CSV / JSON / HTML / ZIP exports (but available via "Export Suppressed (NOISE) CSV").
  5. Mask Value dots out the Value column for screenshots; Mask URL covers both URL and Target columns.

6. Detection Engine Deep Dive

6.1 Pipeline

For each fetched response body, the scanner runs (in tier-aware order):

  1. Form-encoded pre-pass (OAuth-style bodies)
  2. HTML inline-script extraction (recursing as JS)
  3. JSON_SCRIPT_TAG (__NEXT_DATA__ etc.)
  4. Headers-blob path (REQ-HEADERS)
  5. Anchored vendor tokens (Phase 1)
  6. URL credentials (Phase 2)
  7. Custom rules
  8. DB connection strings + context-gated rules (LIGHT, FULL)
  9. GENERIC_KV — full text, no truncation (FULL)
  10. SSR state blobs + high-entropy values + getter functions (FULL)
  11. JWT severity adjustment (HIGH in JSON, MEDIUM in JS)
  12. Two-pass deduplication (line-aware)

6.2 GENERIC_KV — The Workhorse

The single most-fired rule. Matches "key" : "value" or key=value patterns where the key is semantically a credential name and the value passes entropy / shape checks. Runs on the entire text body — no truncation — because Angular / webpack bundles routinely exceed 5 MB and middle-of-bundle keys must not be sampled out.

Common findings:

  • "subscription_Key": "<hex>" → HIGH
  • "appKey": "<base64>" → HIGH
  • "resource": "<uuid>" → HIGH
  • "clientId": "<uuid>" → INFORMATION
  • "tenantId": "<uuid>" → INFORMATION

6.3 Why Findings Repeat at Different Lines

v1.7.1 introduced line-aware deduplication. When the same key+value appears at lines 14, 15, 33, 35, 5321, 5335, … 169754 in a multi-MB Angular bundle, each is a real textual occurrence — not a duplicate. Burp reports them all, and so does SecretSifter. Pass 1 dedup keys on (keyName, value, sourceUrl, line), not just (keyName, value, sourceUrl).

6.4 Why an Anchored Rule Can "Win" Over GENERIC_KV

If GOOGLE_KEY_001 fires at line 42 with value AIzaSy… and GENERIC_KV also fires at line 42 with key apiKey and the same value, dedup Pass 2 keeps GOOGLE_KEY_001 and drops GENERIC_KV. The anchored rule carries the vendor classification ("Google API Key") which is more useful than a generic key label.

8. Triage Guide & FP Determination

8.1 Validating a Finding

  1. Read the Context field — is the value in a real-looking config object (e.g. { apiKey: "...", baseUrl: "https://api...." })?
  2. Click the Source URL link — does the file actually contain the value at the reported line?
  3. Open the Request / Response tabs to see the surrounding HTTP transaction.
  4. If the value is a UUID, check whether it's a public client ID (Microsoft B2C apps, Auth0 client IDs, etc.) vs a secret.
  5. Test the credential out-of-band where applicable (e.g., curl -H "Ocp-Apim-Subscription-Key: ..." <api-base>).

8.2 Common False-Positive Shapes

PatternWhy it's likely FP
"apiKey": "FirstName"Schema field name in a CRUD form, not a credential. SecretSifter rejects all-alpha values; if you see one, please report it.
"clientID": "00000000-0000-0000-0000-000000000000"Placeholder UUID; isPlaceholder() should reject it.
"key": ".bs.tab" or ".data-api"jQuery / Bootstrap event namespace, not a credential — filtered.
"apiKey": "${ENV_VAR}"Template placeholder — filtered.
X-CSRF-Token in headersAnti-forgery token, public — filtered.
Public Microsoft B2C client IDsReported as INFORMATION because they're public, not secret. Mark as NOISE if irrelevant.

8.3 When to Mark NOISE

Set Severity or Confidence to NOISE when:

  • You've manually verified the value is a public identifier or test fixture.
  • The finding is a placeholder you can't reproduce in the wild.
  • Your scope explicitly excludes this kind of finding (e.g., a pentest scope that excludes informational disclosures).

NOISE findings are still visible (greyed) so you can audit your triage later, and exportable separately via Export Data → Export Suppressed (NOISE) CSV.

10.2 PowerShell Smoke Test

$port  = 8765
$token = "<paste from Settings → AI / MCP>"
$h     = @{ Authorization = "Bearer $token" }

# Health
Invoke-RestMethod -Uri "http://localhost:$port/mcp" -Headers $h

# Get findings
Invoke-RestMethod -Uri "http://localhost:$port/api/findings" -Headers $h |
    Format-Table severity, ruleId, keyName, value -AutoSize

# Start a scan
$body = @{ urls = @("https://example.com") } | ConvertTo-Json
Invoke-RestMethod -Uri "http://localhost:$port/api/scan" -Method POST -Headers $h `
    -Body $body -ContentType "application/json"

10.3 Python Snippet

import requests
PORT = 8765
TOKEN = "..."
h = {"Authorization": f"Bearer {TOKEN}"}

# start scan
requests.post(f"http://localhost:{PORT}/api/scan",
              headers=h, json={"urls": ["https://example.com"]})

# poll until idle
while requests.get(f"http://localhost:{PORT}/api/scan/status", headers=h)
        .json()["running"]:
    time.sleep(2)

# fetch findings
findings = requests.get(f"http://localhost:{PORT}/api/findings", headers=h).json()

10.4 Claude Code Integration

From Settings → AI / MCP, click Copy CLI Command and run it in a terminal:

claude mcp add --transport http secretsifter-desktop \
    http://localhost:<port>/mcp \
    --header "Authorization: Bearer <token>"

Claude can then start scans, read findings, and export reports through the MCP protocol.

12. Troubleshooting

12.1 Scanner missing findings Burp catches

SymptomLikely causeFix
Subscription key reported once instead of 30+ timesPre-v1.7.1 dedup collapsed across linesUpdate to v1.7.1+; current version is line-aware
Findings missing on multi-MB Angular bundlesPre-v1.7.0 truncated GENERIC_KV input at 1.5 MBv1.7.0+ runs GENERIC_KV on full text
Vapid key not detectedPre-v1.7.0 missing prefixUpdate; vapid is in SECRET_KEY_PREFIXES
Bare-word semantic keys missed (resource)Pre-v1.7.0 required ≥2 segmentsUpdate; single-word allowlist now applies

12.2 Crawler blocked by Akamai / Cloudflare

Symptom: corporate WiFi blocks the scan with "Access Denied" / Akamai error reference. Hotspot works.

  • v1.7.2+ launches real Chrome via setChannel("chrome") with --disable-blink-features=AutomationControlled. Most Akamai bot checks pass.
  • If still blocked: try setting Browser path in Settings to your Chrome install explicitly.
  • Last resort: run the proxy mode and browse manually — once Akamai's bm_sz / _abck cookies are issued, traffic flows through.

12.3 Proxy "not intercepting traffic"

CauseFix
Browser proxy field has comma in portv1.7.4 strips the comma. Use 127.0.0.1:8888 (no comma)
Cert not importedExport CA, import into browser's trusted root authorities
Logs tab shows nothing on intercepted requestsUpdate to v1.7.4+ — every request now logs [Proxy] METHOD url
Site uses HTTP/2 onlyProxy is HTTP/1.1; modern sites may not capture cleanly

12.4 Build fails with WiX exit code 297

Out of disk during MSI link. build/libs/ accumulates one stale ~175 MB JAR per build. Delete or archive every secretsifter-*.jar in build/libs/ except the current version, then retry gradle buildExeInstaller.

12.5 High idle CPU

If SecretSifter shows > 10% CPU when not scanning, check Task Manager → Details. If chrome.exe is the culprit, a Playwright browser leaked from a prior scan — restart the app. If it's SecretSifter.exe / javaw.exe, please file a report with the version and OS build.

14. Building from Source

14.1 Prerequisites

  • JDK 17+ with jpackage (bundled in JDK 14+).
  • WiX Toolset 3.11+ on PATH (for the EXE installer).
  • Gradle (the wrapper script gradlew.bat is bundled).

14.2 Build Steps

# Obfuscated fat JAR (~1m)
gradlew reinsertObfuscatedClasses

# Portable app-image (~2m)
gradlew buildAppImage

# Full Windows EXE installer (~3m)
gradlew buildExeInstaller
Stale-JAR trap: delete every secretsifter-<version>.jar in build/libs/ except the current version before each buildExeInstaller run, otherwise jpackage bundles them all into the installer image and WiX light.exe may fail with exit code 297 on systems with low disk space.

Data Handling & Local Storage

SecretSifter stores all configuration (settings, custom rules, MCP token, AI provider key) locally in the Windows registry under HKCU\Software\JavaSoft\Prefs\com\secretsifter and never transmits it to any external service. Captured HTTP request / response bodies live only in process memory inside the bounded CaptureStore (max 500 entries, 2 MB per body) and are discarded on exit. The MCP / REST server binds to localhost only and is not exposed to the network.
